0 PROJECT OVERVIEW

What Is This Project About?

We build a machine learning system that predicts which forest areas in the Western Ghats are most vulnerable to fire. Using 22 years of satellite data (2003–2025), we train an ensemble of 5 different ML models that work together, each learning different fire patterns. The system tells us: Where fire will occur (location), How much area will burn (extent), and How intense it will be (intensity).

🌿 Current Target: Western Ghats — UNESCO World Heritage biodiversity hotspot spanning 1,600 km along India's west coast (Gujarat to Tamil Nadu).

2 TEMPORAL DESIGN

22-Year Timeline — Capturing ENSO Cycles

Prof. Kiran required 2–3 decades of data to capture climate influence. ENSO (El Niño/La Niña) cycles every 4–5 years, affecting Indian monsoon rainfall and fire. With 22 years we capture 4–5 complete ENSO cycles.

ENSO — The Fire-Climate Link

El Niño: Warm Pacific → reduced Indian monsoon → drought → MORE fires
La Niña: Cool Pacific → enhanced monsoon → wet → FEWER fires

We use the Oceanic Niño Index (ONI) as a direct feature with multi-year lags (6, 12, 24 months) to capture the delayed drought-fire cascade.

Prof. Kiran + Novel
⏱️

Why Temporal Split (Not Random)?

  • No future leakage — 2022 rain can't help predict 2018 fire
  • Tests true forecasting — trained on past, evaluated on future
  • 5-year test block spans ~1 complete ENSO cycle
  • 22 years × 12 months = 264 monthly time steps per pixel

4 PREPROCESSING

Data Cleaning & Preparation Pipeline

Raw satellite data needs careful preprocessing before ML. Each step has a clear purpose — removing bias, handling imbalance, and ensuring model-ready features.

1

Spatial Thinning

Remove spatial autocorrelation by ensuring minimum 1 km distance between any two points. 50 bootstrap iterations — keep the iteration that retains the most points. Prevents the model from "memorizing" nearby pixel clusters.

[Doc1 — Sarkar et al.]
2

Non-Fire Point Sampling

Randomly sample non-fire points from forested areas with zero fire history (2003–2025). Ratio 1:3 (fire:non-fire) — reflects real-world rarity. Minimum 5 km from any fire point. Same forest type as fire points.

[Doc2 — Shahzad et al.]
3

Multicollinearity Screening

Step A: Spearman correlation — remove one from pairs with |ρ| > 0.75
Step B: Iterative VIF — remove highest VIF until all VIF < 10
Keeps only independent, informative features (~20 → 14–16).

[Doc1, Doc2, Doc3]
4

SMOTE — Synthetic Oversampling

Generate synthetic fire samples to balance the training set. Applied ONLY on training data — never on test/validation (prevents data leakage). Target ratio after SMOTE: 1:1.5 (slight underrepresentation to avoid synthetic noise).

x_synthetic = x_i + λ × (x_nearest_neighbor − x_i), λ ~ Uniform(0,1)
[Doc2 — Shahzad et al.]
5

Feature Scaling — StandardScaler

Normalize all features to zero mean, unit variance. Fitted on training set only — same transform applied to test set (prevents information leakage).

x' = (x − μ_train) / σ_train
[Doc2, Doc3]

⏳

Temporal Block Features (~15 features)

Since data is in 5-year temporal blocks, we create block-to-block temporal signals:

  • Previous block values: For each pixel, attach Block(N-1) values as lag features when training Block N
  • Block-to-block deltas: ΔNDVI, ΔTemp, ΔPrecip between consecutive blocks
  • Trend direction: Is temperature rising/falling across blocks?
  • Cumulative fire history: How many times did this pixel burn in prior blocks?
  • ENSO block means: Average ONI value for each block period (from NOAA)
[Novel + Prof. Kiran's temporal directive]
🌏

Climate Indices (~5 features)

External data from NOAA — not available in GEE:

  • ONI (Oceanic Niño Index): Block-mean ENSO state
  • ONI extremes: Max/min ONI during each block period
  • IOD (Indian Ocean Dipole): Block-mean IOD value
  • El Niño month count: Number of months with ONI > +0.5 per block
  • La Niña month count: Number of months with ONI < −0.5 per block
[Prof. Kiran + Novel]
🗺️

Spatial Neighbor Features (~6 features)

What's happening in nearby pixels:

  • Neighbor fire rate: % of k-nearest pixels that burned
  • Local Moran's I: Spatial autocorrelation of fire
  • Mean neighbor NDVI/FRP: Average vegetation & fire intensity of neighbors
  • Spatial fire density: Fire count within R km radius
  • Distance to nearest fire: How close is the nearest fire pixel?
  • Wind-aligned fire spread: Risk from upwind neighbor fires
[Novel Contribution]
🔧

Derived & Encoded Features (~10 features)

Statistical transformations of existing GEE features:

  • Land cover one-hot: Encode IGBP class into multiple binary columns
  • Aspect encoding: sin(aspect) + cos(aspect) for circular variable
  • VPD × dry_months: Additional drought-fire interaction
  • Elevation bands: Categorical (lowland/mid/highland)
  • Fire-weather composite: Combined fire danger score
  • Feature ratios: temp_range/precip, NDVI_range/EVI, etc.

💡 Why Two Stages?

GEE is powerful for satellite data processing at planetary scale — extracting pixel values, computing temporal means, band math. But it can't do ML-specific operations like spatial k-NN, block-to-block deltas, or external data merging.

Python handles the ML-specific feature engineering: temporal context between blocks, spatial relationships between pixels, external climate indices (ONI/IOD from NOAA), and encoding transformations. This two-stage approach is standard in remote sensing ML pipelines.

6 DRIVER WEIGHTING

Learnable Group Weights per Zone

Not all feature groups matter equally in every zone. We learn softmax weights for 4 driver groups — each zone gets its own set of weights.

How It Works

[α_weather, α_fuel, α_topo, α_human] = softmax(w), w ∈ ℝ⁴

x_weighted = α_w · x̃_weather ∥ α_f · x̃_fuel ∥ α_p · x̃_topo ∥ α_h · x̃_human

Bayesian Optimization (Optuna, 50 trials per zone) finds optimal weights by maximizing validation AUC.
SHAP-derived weights used as sanity check: α_g ∝ mean|SHAP_g|

Total: 4 weights × 3 sub-zones = 12 parameters — negligible overfit risk.

🔄 Cross-Validated Meta-Features

K=5 stratified folds (stratified by zone + class). Each base model is trained on 4 folds, predicts on the held-out fold. This generates unbiased "meta-features" for the meta-learner — zero data leakage.

p̂_i^(b) = f_b^(−κ(i))(x_weighted,i)
[Doc2, Doc3]

🤔 Why These 5 Models?

  • RF: Robust, handles noise, OOB error estimation
  • XGBoost: Regularized boosting — top performer in Doc2 & Doc3
  • SVM: Kernel-based decision boundary — different geometry
  • MLP: Neural network captures non-linear interactions
  • LightGBM: Leaf-wise growth — different from XGBoost's level-wise

Diversity is key — models that make different errors combine better.

8 POST-PROCESSING

Uncertainty Quantification & Interpretability

Predictions alone aren't enough — we need to know how confident the model is and why it made each prediction. This makes the system trustworthy for real-world use.

🛡️

Conformal Prediction — Guaranteed Uncertainty

A distribution-free method that provides prediction sets with guaranteed coverage:

s_i = 1 − p̂(y_i) (nonconformity score)
q̂ = Quantile({s_i}; ⌈(n+1)(1−α)⌉/n)
C(x_new) = {c ∈ {0,1} : p̂(c) ≥ 1 − q̂}

Coverage guarantee: P(Y_new ∈ C(X_new)) ≥ 1 − α for ANY model and ANY distribution.
Calibrated on 2019–2020 data (8% of total).

Novel Contribution
🔍

SHAP — Why Did the Model Predict This?

TreeSHAP for interpretability at 4 levels:

  • Group-level: Which driver group matters? (Weather vs Fuel vs Topo vs Human)
  • Feature-level: Which specific variable? (Temperature? NDVI? Slope?)
  • Temporal-level: Which lag period? (ONI-lag-12 vs temp-lag-3)
  • Zone-level: How do top features differ between Z1 vs Z2 vs Z3?
[Doc2, Doc4]

Source 1: Conformal

Distribution-free prediction sets. Small sets = confident. Large sets = uncertain.

Source 2: MC Dropout

Run MLP 50 times with random dropout. High variance in predictions = model uncertainty.

Var = (1/T)Σp̂² − ((1/T)Σp̂)²

Source 3: Ensemble Disagreement

If the 5 base models disagree, the prediction is uncertain.

Var = (1/5)Σ(p̂_b − p̄)²
